## Problem-Solving Strategy

### How to Think About Any Problem

#### Step 1: Understand the Problem (5 minutes)
```
1. Read the problem statement 2-3 times
2. Identify INPUT and OUTPUT
3. Write down constraints
4. Create your own test cases:
   - Normal case
   - Edge case (empty, single element)
   - Large case (maximum constraints)
```

**Example**: "Find the maximum sum subarray"
```
Input: Array of integers (can be negative)
Output: Maximum sum (single number)
Constraints: Array size 1 to 10^5, values -10^9 to 10^9

Test cases:
Normal:  [1, 2, 3, -2, 5] → 9
Edge:    [-5] → -5
All neg: [-3, -2, -1] → -1
Mixed:   [2, -5, 3, 1, -2, 4] → 6
```

#### Step 2: Brute Force First (10 minutes)
```
1. Think of the simplest solution
2. Don't worry about efficiency
3. Write pseudo-code
4. Check if it gives correct output
```

**Why start with brute force?**
- Guarantees you have A solution
- Helps understand the problem deeply
- Gives baseline for optimization
- Better than no solution in interviews!

#### Step 3: Optimize Step-by-Step (10-15 minutes)
```
Ask yourself:
1. Are we doing redundant work?
2. Can we store and reuse results?
3. Is there a pattern in the data?
4. Can we sort/preprocess?
5. Does a data structure help?
```

**Optimization Techniques**:
- Repeated calculations? → Use Hashing or DP
- Need min/max in range? → Sliding Window
- Sorted data or can sort? → Binary Search
- Pairs/Triplets? → Two Pointers
- Top K elements? → Heap
- Subsequences? → DP or Recursion

#### Step 4: Code Cleanly (10 minutes)
```
1. Use meaningful variable names
2. Write helper functions
3. Handle edge cases first
4. Add comments for complex logic
5. Test with your test cases
```

#### Step 5: Debug Systematically
```
1. If wrong answer, dry run on small case
2. Print intermediate values
3. Check boundary conditions
4. Verify edge cases
5. Compare with brute force on small inputs
```

### Pattern Recognition Framework

**If you see these keywords, think these patterns**:

| Problem Keywords | Likely Pattern |
|------------------|----------------|
| "Sorted array", "Find element" | Binary Search |
| "Top K", "Kth largest/smallest" | Heap/Priority Queue |
| "Maximum/Minimum subarray" | Sliding Window or Kadane's |
| "All permutations", "All subsets" | Recursion/Backtracking |
| "Shortest path", "Minimum steps" | BFS |
| "Longest common", "Maximum ways" | Dynamic Programming |
| "Cycle detection", "Connected" | Graph (DFS/BFS/Union-Find) |
| "Pairs with sum X" | Two Pointers or Hashing |
| "Valid parenthesis", "Next greater" | Stack |
| "Prefix", "Range sum" | Prefix Sum or Segment Tree |

### How to Break Down Unseen Problems

**The 5-Step Framework**:

1. **Simplify**: Can you solve a smaller version?
   - If array of size N, try N=1, N=2, N=3
   - Identify what changes and what stays constant

2. **Visualize**: Draw it out!
   - Arrays: Draw boxes with indices
   - Trees/Graphs: Sketch nodes and edges
   - Process: Show step-by-step transformation

3. **Classify**: What type of problem is this?
   - Searching? → Binary Search, BFS
   - Optimization? → DP, Greedy, Binary Search on Answer
   - Counting? → Combinatorics, DP
   - Construction? → Greedy, Backtracking

4. **Relate**: Have you seen something similar?
   - Same data structure?
   - Same pattern with different context?
   - Can you transform this to a known problem?

5. **Iterate**: Start simple, then improve
   - Brute force → Better → Optimal
   - Each step teaches you something

## Interview Strategy Guide

### FAANG Interview Process

**Typical Structure** (45-60 minutes):
```
0-5 min:   Introduction and small talk
5-10 min:  Problem statement and clarification
10-25 min: Thinking and discussing approach
25-40 min: Coding the solution
40-50 min: Testing and edge cases
50-60 min: Complexity analysis and follow-ups
```

### Communication Framework During Interviews

**What Interviewers Evaluate**:
1. Problem understanding (Do you ask good questions?)
2. Thought process (Can you think out loud?)
3. Approach selection (Do you consider trade-offs?)
4. Coding skills (Clean, bug-free code?)
5. Testing (Do you verify your solution?)
6. Complexity analysis (Do you know Big-O?)

**Step-by-Step Interview Strategy**:

#### 1. When Problem is Given (First 2 minutes)
```
DO:
- Read carefully, don't rush
- Ask clarifying questions:
  * "Can the input be empty?"
  * "What's the range of values?"
  * "Should I handle invalid input?"
  * "Are there duplicates?"
- Repeat the problem in your own words
```

**Example Script**:
> "So, if I understand correctly, I need to find [X] given [Y]. 
> The input will be [type], and I should return [type].
> Just to confirm, [edge case question]?"

#### 2. Before Coding (5-10 minutes)
```
DO:
- Think out loud (interviewers want to hear your thoughts)
- Mention brute force first
- Discuss optimization ideas
- Get buy-in before coding:
  "I'm thinking of using [approach] because [reason]. 
   This will give us O(X) time and O(Y) space. 
   Does this sound good?"
```

#### 3. While Coding (15-20 minutes)
```
DO:
- Explain what you're doing
- Write clean, readable code
- Use meaningful variable names
- Handle edge cases explicitly
- Don't stay silent for more than 30 seconds
```

**Say things like**:
> "I'm creating a hash map to store frequencies..."
> "Now I'll iterate through the array..."
> "This edge case occurs when the array is empty, so I'll handle it first..."

#### 4. After Coding (5-10 minutes)
```
DO:
- Walk through your code with an example
- Test with edge cases yourself
- State time and space complexity
- Discuss trade-offs
- Mention alternative approaches
```

**Example**:
> "Let me trace through with input [1, 2, 3]:
> - Step 1: ...
> - Step 2: ...
> - Final output: ...
> 
> The time complexity is O(n) because we iterate once.
> Space complexity is O(n) for the hash map.
> We could optimize space to O(1) if we use two pointers, 
> but that would require sorting first, making it O(n log n)."

### Common Interview Patterns to Master

**Must-Know Patterns** (in order of frequency):
1. Two Pointers (15-20% of interviews)
2. Sliding Window (10-15%)
3. Hash Map/Set (15-20%)
4. BFS/DFS on Trees/Graphs (15-20%)
5. Dynamic Programming (10-15%)
6. Binary Search (10%)
7. Heap/Priority Queue (5-10%)
8. Backtracking (5-10%)

## Competitive Programming Guide

### Platform Recommendations

**For Beginners**:
- **HackerRank**: Great for learning basics, topic-wise practice
- **LeetCode**: Best for interview prep, good community
- **CodeChef (Starters)**: Beginner-friendly contests

**For Intermediate**:
- **LeetCode (Contest)**: Weekly contests, rating system
- **CodeChef**: Long challenges, good for learning
- **AtCoder (ABC)**: Excellent problem quality

**For Advanced**:
- **Codeforces**: Best for CP, strong community
- **AtCoder**: Mathematical problems, great for thinking
- **SPOJ**: Classic problems, challenging

### Contest Strategy

**Before Contest**:
- Sleep well, stay hydrated
- Have a quiet environment
- Prepare template code (fast I/O, common functions)

**During Contest** (2-hour contest):
```
First 10 minutes:
- Read ALL problems
- Identify easiest problems (usually A, B)
- Plan your approach

Next 90 minutes:
- Solve in order of difficulty
- Don't get stuck on one problem (>30 min = move on)
- Submit quickly (penalties are small, learning is big)

Last 20 minutes:
- Revisit unsolved problems
- Double-check submissions
- Try to squeeze in one more solution
```

**After Contest**:
- Upsolve problems you couldn't solve
- Read editorials and top solutions
- Learn new techniques
- Add to your template

### Time Management in Contests

**Problem Selection Priority**:
1. Solve what you KNOW first
2. Then attempt what you CAN figure out
3. Leave hardest for last (or skip)

**Time Allocation** (2-hour contest, 6-8 problems):
- Easy (A, B): 10-15 min each
- Medium (C, D): 20-30 min each
- Hard (E, F, G): 40+ min each (if you have time)

```

   ↓ (Understand the concept)
   
2. ATTEMPT practice.md
   ↓ (Struggle is good! Try for 20-30 min)
   
3. CHECK hints (if stuck)
   ↓ (Don't peek at solution yet!)

   ↓ (Implement yourself)
   
5. COMPARE with solutions.md
   ↓ (Learn better approaches)
   
6. TAKE mcqs.md
   ↓ (Test your understanding)
   
7. REVISION (after 1 day, 3 days, 1 week)
   ↓ (Spaced repetition)

```

### Rules for Success

**DO**:
- Solve problems YOURSELF before looking at solutions
- Write code by hand (improves thinking)
- Keep a mistake journal
- Revise regularly (spaced repetition)
- Participate in contests
- Teach others (best way to learn)

**DON'T**:
- Copy-paste solutions
- Jump to advanced topics without basics
- Skip revision
- Only solve easy problems
- Give up after one failed attempt
- Compare your progress with others

Keep a separate notebook or file:


## Revision Strategy & Spaced Repetition

### The Forgetting Curve

Research shows we forget:
- 50% within 1 hour
- 70% within 24 hours
- 90% within 1 week

**Solution**: Spaced repetition at optimal intervals

### Revision Schedule

```
Learn Topic → Revise at:
- 1 day later (quick review)
- 3 days later (solve 2-3 problems)
- 1 week later (solve 5 problems)
- 2 weeks later (mixed practice)
- 1 month later (mock test)
```

### Weekly Revision Template

**Sunday (Revision Day)**:
```
Morning (2 hours):
- Revise topics learned this week
- Re-solve problems you got wrong
- Take MCQs from completed topics

Evening (1 hour):
- Solve 5 mixed problems (random topics)
- Participate in contest (if available)
- Update mistake journal
```

### Revision Checklist per Topic

```
□ Can I explain this concept to someone else?
□ Can I solve Level 1-2 problems without help?
□ Can I solve at least 50% of Level 3 problems?
□ Did I score 80%+ on MCQs?
□ Can I identify when to use this pattern?
□ Have I solved at least 15-20 problems on this topic?
```
